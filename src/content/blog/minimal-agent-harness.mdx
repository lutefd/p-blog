I wanted to understand how little an agent harness actually needed.

Not how little a demo needed. Demos are easy. A chat box, a model call, a tool or two, some streaming text, and the whole thing looks convincing enough for a screenshot.

I mean the harness underneath the work.

The part that keeps the session alive. The part that decides when tools run. The part that shows what happened. The part that handles provider events, user interruption, credentials, local state, approvals, and all the weird little boundaries that get hand-waved when people say "just give the model tools."

I had two reasons for caring.

The first was that I was learning more about prompt injection. The more I read and experimented, the less satisfied I was with vague advice like "be careful what you put in context" or "sandbox your tools." Those are true, but they are not the interesting part. The interesting part is where the boundary actually lives.

What does the model get to see?

What does the tool runner trust?

Can a tool result change the next request?

Can an extension block a tool call before it runs?

Can a provider adapter ask the client to do something?

Which parts are policy, which parts are UI, and which parts are just local execution?

The second reason was less theoretical.

I was tired of waiting on Claude Code.

The tool was useful, but the experience often felt sluggish in the exact place where an agent should feel sharp: the turn loop. I would ask for something, wait through the machinery, watch a tool run, wait again, and feel too far away from the system doing the work.

I did not want to clone the whole product.

I wanted to poke at the loop.

## The Turn Loop Is The Product

The more I used agent tools, the more I felt that the "agent" was not the model. The model mattered, obviously, but the product lived around it.

The product was the turn loop.

A user writes something. The system builds a request. The model streams text or tool calls. The local runtime decides whether to execute them. Results come back. The model continues. The user watches, interrupts, approves, retries, or changes direction.

That loop has texture.

If the interface hides too much, the user stops trusting it. If the tool output is too noisy, the model and the person both drown in it. If cancellation is slow, the system feels dangerous. If provider configuration is precious, experimentation gets expensive. If every extension idea requires changing core code, the harness becomes a pile of special cases.

So I started thinking in smaller units.

What is the smallest terminal surface that still feels usable?

What is the smallest provider abstraction that can stream text, reasoning, and tool calls?

What is the smallest tool protocol that can support normal shell commands today and more structured interaction later?

What is the smallest extension model that can intercept the loop without taking over the whole app?

This was not about minimalism as an aesthetic. It was about being able to see the system clearly enough to learn from it.

## A Harness, Not An IDE

The first useful decision was to keep the thing close to the terminal.

An IDE wants to own a lot of space. It has panels, files, tabs, decorations, language servers, and all the affordances of a place where you live for hours. That is useful, but it was too much surface area for the experiment I wanted.

A terminal harness has a different constraint. It has to be direct. It has to show enough state without becoming a dashboard. It has to make common actions cheap: start a session, resume a session, switch models, open the inspector, stop the turn, reload extensions, check provider health.

That led to the basic shape:

- a streaming TUI;
- resumable sessions;
- built-in read, write, edit, shell, and tool-listing tools;
- model switching;
- an inspector pane;
- reloadable runtime assets;
- keychain-backed credentials;
- a machine-readable RPC mode for other clients.

None of those are impressive on their own.

Together, they make the harness feel less like a wrapper and more like a place where the loop is visible.

That visibility mattered more than I expected. When a tool is slow, you want to see the tool. When the model is thinking, you want to know the turn is alive. When something breaks, you want diagnostics near the runtime, not hidden behind a polished chat transcript.

## Prompt Injection Made The Extension Model Sharper

Prompt injection is easy to discuss badly because it turns into a giant warning label.

Do not trust untrusted text. Do not let tool output override system instructions. Do not leak secrets. Do not blindly execute what a page tells the model to execute.

Again: true, but not enough.

The practical question is where a local agent runtime can put seams.

Sometimes you want to transform user input before it reaches the model. Sometimes you want to add local context to the prompt. Sometimes you want to inspect a tool call before execution and block it if it violates a policy. Sometimes you want to patch a tool result before it gets persisted. Sometimes you only want an async side effect after the assistant finishes.

Those should not all be the same mechanism.

The extension model started to separate into surfaces:

- tools for isolated local actions;
- hosted tools for stateful actions owned by a long-lived process;
- providers for model transports;
- UI commands and inspector views;
- approval policies;
- async hooks;
- prompt additions;
- skills;
- themes;
- long-lived extension hosts for sync interception and session state.

The rule of thumb became simple:

If the behavior is static registration, use a manifest.

If it needs session state, sync interception, or hosted tool execution, use a long-lived extension host.

If it is a side effect, keep it async.

If it is provider translation, make it a provider adapter.

This kept the core smaller. More importantly, it kept the boundaries visible. A hook should not quietly mutate the active turn. A prompt extension should not become a tool policy. A provider adapter should translate model events, not secretly own local execution.

The point was not to create a platform for its own sake.

The point was to keep the experiment from turning into mud.

## Providers Should Not Be Sacred

Another thing I wanted was to make providers boring.

Most agent tools start with one blessed provider path, then slowly add conditionals. This model has tool calls. This one streams differently. This one has reasoning events. This one needs a gateway. This one is local. This one is OpenAI-compatible except for the part where it is not.

I wanted the local runtime to treat provider transport as replaceable.

An OpenAI-compatible provider can be configuration: base URL, API key environment variable, model list. A custom provider can be an exec adapter. The adapter receives one JSON request on stdin and streams provider events back as JSONL: thinking, text deltas, tool calls, client actions, done.

Tool execution still belongs to the local runtime.

That division is important. The provider can ask for a tool call, but the harness decides how local tools are represented, approved, executed, logged, and returned. The adapter translates the upstream API into the local protocol. It does not become the whole application.

This made it much easier to experiment with local gateways, custom model APIs, and provider-specific behavior without rebuilding the app around each one.

It also made the security questions cleaner. Provider events are not the same thing as local authority. A streamed tool call is a request. The harness still has to decide what happens next.

## Local Layers Made Experimentation Cheap

The runtime needed a place to put experiments.

Global configuration is useful, but project-specific behavior matters a lot for agent work. One repository may need custom tools. Another may need a different provider. Another may want stricter approval policies. Another may carry skills that explain local workflows.

So the lookup model became layered:

1. user global runtime files;
2. user-installed packages;
3. project-installed packages;
4. project overrides.

Later layers override earlier ones.

That means a tool can live in `~/.luc`, a package can bring a provider or theme, and a workspace can override prompts, approvals, or tools without changing the binary. `luc reload` picks up runtime changes without restarting the whole app.

This is where the harness started feeling useful beyond the original experiment. A small runtime becomes much more flexible when it can be reshaped per workspace.

Not every team needs the same agent.

Not every repository should expose the same tools.

Not every experiment should wait for core code.

## Then It Became Luc

That is when the experiment got a name.

Luc is the terminal agent harness I built around these constraints: keep the loop local, make the runtime visible, treat providers as adapters, and let tools, prompts, hooks, UI, skills, themes, and extension hosts load without recompiling.

The name matters less than the shape.

The shape is intentionally small:

```bash
luc
luc open <id>
luc doctor
luc reload
luc rpc
```

The commands are not the interesting part.

The interesting part is what they imply. A session can be resumed. Runtime assets can be reloaded. Provider and workspace state can be inspected. The same harness can be driven by a human TUI or machine-readable RPC.

The app also keeps the boring everyday details close: credentials can come from the environment or the OS keychain; project config can override user config; built-in tools cover the basic file and shell loop; runtime packages can add more surfaces.

I wanted something that made the agent loop feel close enough to touch.

That is still the goal.

## What I Learned

The first thing I learned is that small harnesses are clarifying.

When the runtime is small, design mistakes are harder to hide. If cancellation feels bad, you feel it. If tool output is too noisy, you see it. If a provider abstraction is wrong, every streamed event reminds you. If the extension model is too broad, the whole system starts to feel unsafe.

The second thing I learned is that extensibility needs sharper boundaries than I first expected.

"Let users extend it" is not a design. Extend what? At what point in the turn? With what authority? Does it block? Can it mutate? Can it ask the client to do something? Does failure stop the session or become a diagnostic?

Those questions are the product.

The third thing I learned is that sluggishness is not only performance.

Sometimes a tool feels slow because it is slow. Sometimes it feels slow because you cannot see what it is doing, cannot interrupt it cleanly, cannot reload the thing you just changed, or cannot tell whether the delay is the model, the provider, the tool, or the UI.

A good harness should make those distinctions visible.

## Where This Goes

I do not know if this becomes a serious daily driver, a testbed, or just the place where I keep learning how agent runtimes should be shaped.

That is fine.

The useful thing is the experiment.

Can a terminal-first harness stay small while supporting real extension points?

Can provider adapters stay replaceable without flattening every model into the same lowest-common-denominator stream?

Can prompt-injection-aware seams be explicit enough that tools, context, and user authority do not blur together?

Can a local agent feel fast because the loop is visible, not because the UI pretends nothing complicated is happening?

Those are the questions I wanted a place to ask.

The answer, so far, is that the harness matters more than I thought.
