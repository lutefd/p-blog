I like stacked branches, but I dislike how often the workflow feels like it belongs to the review platform instead of the repository.

The moment the stack is defined by PR metadata, a weird dependency appears. The work exists locally, the branches exist locally, Git can already express the history, but the shape of the work only becomes understandable after it has been pushed somewhere else.

That is backwards for the way I want to work.

Weaver starts with a smaller bet: a branch stack is local orchestration state. It should work before a PR exists, while a PR is changing, and after remote metadata gets stale.

## Declare, Inspect, Then Mutate

The first thing Weaver does is refuse to make the stack magical.

It stores dependency relationships in `.git/weaver/`, keeps them out of the committed tree, and resolves stacks as a DAG. `weaver stack` and `weaver unstack` declare parent-child relationships. `weaver deps`, `weaver status`, and `weaver doctor` show the stack and diagnose broken local state.

Only after that do commands like `weaver update`, `weaver sync`, and `weaver compose` start moving branches around.

That order is the point. Stacked work gets scary when a tool hides the model. Weaver keeps the model plain: dependencies, upstream refresh, sync strategy, and integration output.

## Rebase, Merge, Or Compose

The project also comes from a practical frustration: different stack operations get described with the same vague words.

`weaver update` fetches upstream refs and fast-forwards selected local branches. It does not sync a branch with its stack parent.

`weaver sync` brings a stack back into dependency order. By default it uses rebase for clean stacked diffs, but `weaver sync --merge` preserves branch history when open PRs, review comments, or other consumers make rewriting awkward.

`weaver compose` is a different thing again: it builds an ephemeral integration state from several branches on top of a base. It can also create or update a named integration branch, and saved integration strategies live in `.git/weaver/integrations.yaml`.

Separating those jobs makes the workflow easier to reason about. Refreshing from upstream is not the same as syncing a stack, and syncing a stack is not the same as testing the combined result.

## The Safety Model Is The Product

Git tools earn trust in the boring details.

Weaver prints mutating Git commands before execution. Sync state is persisted before each step. `weaver abort` restores the original branch. Compose is ephemeral by default. Failures name the branch and conflicting files. Integration branches created by compose can be listed and deleted later.

That is the right taste for this kind of tool. The value is not hiding Git. The value is making the local orchestration inspectable enough that a developer can trust it when history gets awkward.
