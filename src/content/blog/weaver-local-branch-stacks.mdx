Stacked branches are usually presented as a code review feature. That framing is useful, but it hides the more interesting primitive: a stack is a local history shape.

Weaver starts from the local shape first. A developer should be able to split work into reviewable layers, understand parent-child relationships, rebase the stack, and move through the work without depending on GitHub metadata as the source of truth.

## Why Local First

Git already has the objects. The branch host has review state, comments, checks, and collaboration affordances, but it should not be required to understand the stack itself.

Local-first stack management keeps the workflow available before a PR exists, while a PR is changing, and after remote metadata gets stale.

## The Workflow Texture

The ideal stack tool should make small actions cheap:

- see the current stack,
- create the next branch,
- restack after changes,
- land work in order,
- recover when history gets awkward.

The hard part is not inventing more Git concepts. The hard part is making the existing ones feel less sharp.

## What Weaver Is For

Weaver is an attempt to make stacked work feel like ordinary Git rather than a separate review product. It should help with the mechanical parts of branch relationships while staying honest about the local repository as the source of truth.
