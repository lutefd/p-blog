I started building Luc because I wanted to understand how little an agent harness actually needed.

At the time, I was learning more about prompt injection and the weird security boundary around tools, context, and model output. The more I looked at it, the more I wanted a small place to experiment. Not a full IDE. Not a product surface with ten layers of magic. Just the loop itself: messages, tools, provider events, approvals, runtime hooks, and enough UI to see what was happening.

There was also a more personal reason.

I was tired of waiting on Claude Code.

The experience was powerful, but it often felt sluggish in the exact place where an agent tool should feel sharp: the turn loop. I wanted to poke at the machinery directly. What happens if the interface is a terminal TUI? What happens if providers are just adapters? What happens if tools are local manifests? What happens if extensions can intercept the loop without becoming the core application?

Luc became my answer to that itch.

## The Harness Before The Product

The first useful framing was to stop thinking of Luc as "my agent app" and start thinking of it as a harness.

An agent harness has a few boring jobs:

- keep a session;
- stream model output;
- run local tools;
- show what the tool is doing;
- let the user stop the turn;
- preserve enough state to resume;
- make the runtime inspectable.

That sounds basic, but basic is where the taste lives. If the harness feels slow, opaque, or overgrown, every experiment built on top of it inherits that weight.

So Luc stayed close to the terminal. It has a streaming TUI, an inspector pane, model switching, a command palette, session reopen, reload, and `luc rpc` for machine-readable mode. Those are not glamorous features, but they make the loop feel like something you can work inside instead of wait beside.

## Why Extensions Became The Center

The more I experimented, the less I wanted every idea to become core code.

Prompt injection work makes this especially obvious. Sometimes you want to transform user input before it reaches the model. Sometimes you want to add hidden context. Sometimes you want to block or patch a tool call before it executes. Sometimes you only need an async hook after a message finishes. Sometimes you need a provider adapter. Sometimes you need a little inspector view.

Those should not all be the same extension point.

Luc's runtime model split them into surfaces:

- exec and hosted tools;
- OpenAI-compatible or exec providers;
- UI commands, views, and approval policies;
- async hooks;
- prompt additions;
- skills and themes;
- long-lived extension hosts for stateful or synchronous behavior.

That distinction matters because it keeps the system small. If the behavior is static, a manifest is enough. If it needs session state or sync interception, it can live in a `luc.extension/v1` host. If it is only a side effect, it can be a hook. If it is provider translation, it can be an exec provider.

The goal was not to design an extension platform because platforms are fun.

The goal was to keep the harness from becoming a junk drawer.

## The Local Runtime Boundary

Luc loads runtime surfaces from user and workspace layers: `~/.luc`, installed packages, and project-local `.luc`. That shape came from wanting experiments to be local and replaceable.

A project can bring its own tools, prompts, providers, approvals, or skills without recompiling Luc. A user can keep global preferences and credentials. A package can ship a bundle of runtime surfaces. `luc reload` picks changes up without restarting the whole app.

This also made provider work less precious. An OpenAI-compatible endpoint is just configuration. A custom model API can be an exec adapter that receives one JSON request and streams JSONL provider events back. Tool execution still happens inside Luc; the adapter only translates the upstream API into the harness protocol.

That was the part I wanted to learn by building instead of theorizing.

Where is the boundary between the model provider and the local tool runner?

Where should approvals live?

Which events need to block the turn loop, and which ones should observe without being able to break the session?

Luc gave me a place to answer those questions with code.

## What I Wanted From The Experience

I did not want another heavy agent product.

I wanted something that made the agent loop feel close enough to touch.

That is why the small details matter: keychain-backed credentials, workspace config, reloadable extensions, built-in file and shell tools, inspector views, hosted tools, client actions, and a TUI that can stop a turn immediately when it goes wrong.

The interesting thing about Luc is not that it talks to models. Everything talks to models now.

The interesting thing is the harness: small enough to experiment with, local enough to trust, and extensible enough that new ideas do not immediately have to become core code.
