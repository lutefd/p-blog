I kept running into a strange failure while using agents on real repositories.

The repository had already answered the question.

The next person or agent doing the work had no way to know that.

Not in the usual "documentation is hard" sense. This was more specific, and honestly more annoying. A reviewer would explain why a behavior belonged at a shared boundary. The PR would change direction. The code would merge. Then, a few weeks later, another task would reach the same edge from a different package and an agent would propose the same locally reasonable shortcut.

The old decision existed somewhere.

It just did not travel.

At first, I thought this was an agent orchestration problem. Agents were going to run in parallel. Developers were going to use more worktrees. Review was going to get messier. Someone would need to coordinate sessions, branches, attempts, logs, and merge flows.

That is probably true.

But it was not the core problem I kept seeing.

The core problem was that software work was becoming more disposable, while repository knowledge was still trapped in the places where it happened.

## The Same Correction, Wearing A New Shirt

The frustrating thing about this failure is that the generated code can look good.

An agent proposes a change. The implementation is tidy. The tests pass. The approach is not obviously dumb.

Then a reviewer says: we do not do it that way here.

Not because of some universal best practice. Because this repository has history. Maybe this behavior belongs in one shared layer. Maybe a package is halfway through a migration. Maybe an earlier incident taught the team not to split one semantic decision across three feature paths.

The PR changes and merges.

But what survives?

The patch survives.

The reason usually becomes folklore.

Git preserves the final shape of the code. The PR preserves the discussion if someone knows where to look. The transcript may survive inside a provider account. But none of that means the next thread receives the relevant decision at the moment it chooses an approach.

So the same correction comes back.

And with agents, it comes back faster.

Parallel work makes it worse. Three worktrees can rediscover the same rejected path at the same time, each producing a polished local solution. The human then has to repeat the same architectural explanation, not because the team failed to decide, but because the decision never became reusable project direction.

That distinction matters.

This is not always a lack of coordination.

Sometimes the coordination happened. The review was clear. The decision was good.

The work just did not inherit it.

## Why A Bigger Prompt File Was Not Enough

The obvious fix is to write more instructions.

Add the rule to `AGENTS.md`. Add it to `CLAUDE.md`. Add it to Cursor rules. Add more examples. Add more warnings. Make the agent read more before it writes.

This works for a while.

Then the instruction file turns into a museum of old pain.

Every rule goes in. Few rules come out. Some apply everywhere. Some apply to one migration. Some were temporary. Some were copied from a review nobody remembers. Some are still important but only under one path or package.

Eventually the useful warning disappears into prompt sediment.

The problem is not just that agents need more context. It is that they need the right context, scoped to the work they are about to do.

A routing decision should not burden a CSS task. A temporary migration constraint should not become permanent law. A review correction about one package should not silently become a global rule.

A global instruction file is not scoped repository memory.

It is a noticeboard.

## Why General Memory Also Felt Wrong

The next obvious answer is memory.

Store conversations. Store PRs. Store review comments. Store docs. Retrieve useful facts later.

That sounds close, but it still felt like the wrong primitive.

A transcript is what happened.

A repository decision is the small part of what happened that should change future work.

Those are very different objects.

Most of a conversation should not become durable project knowledge. It can contain brainstorming, wrong assumptions, temporary plans, stale facts, private context, and long stretches that only mattered inside one task.

Replaying all of that into future work is noisy and expensive. It also still leaves the important question unanswered:

What did we learn that the next person or agent should not have to rediscover?

That question pushed me away from “agent memory” and toward something narrower.

Not memory in general.

Repository direction.

A compact piece of project knowledge with a boundary:

- what was decided;
- where it applies;
- why it applies;
- which path was rejected;
- which path is preferred;
- who or what asserted it;
- whether it is temporary, candidate, or durable;
- what evidence supports it;
- whether later work actually implemented it.

That was the missing shape.

## The Abstraction That Started To Stick

Once I stopped thinking in terms of memory, the design got clearer.

A useful record should be small. It should not contain the whole conversation. It should not contain source code or patches. It should not try to become a second Git history.

It should be more like:

> In this scope, prefer this direction.
> This was the rationale.
> This tempting alternative was rejected.
> This evidence explains why.
> This record is live, candidate, or durable.

Scope became the most important part.

Large repositories do not have one truth. They have local truths. Routing has its own rules. Caching has its own constraints. Build tooling has its own weird edge cases. Product workflows have assumptions that make no sense outside their area.

So the record needed to be scoped by repository, issue, PR, configured area, path, or, rarely, global.

Then a thread could declare what it was working on and receive only the active records that matched that scope.

No semantic guessing required. No embedding search. No model deciding what feels relevant.

Just deterministic projection.

Given the same records, scope, config, receipts, and budget, the same records should be selected.

That felt boring in the right way.

## Availability Is Not Causality

Another thing bothered me while thinking about this: explanation theater.

A lot of systems can say something was “used” because it was somewhere in context.

That is not honest.

If a rule was present in a prompt, that proves availability. It does not prove influence.

A useful “why did this happen?” view needs to separate those claims.

It should be able to say:

This direction existed.

This direction matched the thread scope.

This direction was delivered.

This direction was exposed to the model.

Then, separately:

This action explicitly declared that it was informed by the direction.

Or:

This commit explicitly declared that it implements the direction.

Those last claims should not be inferred from text similarity. They should not be guessed from the final diff. They should be explicit causal relations asserted by an adapter or actor.

That distinction made the system feel less magical, but much more trustworthy.

It also gave me a useful rule:

If the system does not know, it should say unknown.

Not “probably used.”

Not “semantically related.”

Unknown.

## Then It Became Fabric

That is when the thing finally had a name.

Fabric is the local-first protocol I built around this idea: persistent repository direction and causal provenance across agent threads, worktrees, and tools.

The first version is intentionally small and local.

No account. No server. No database. No daemon. No LLM call. No embedding pipeline.

Just immutable protocol events stored in repository-controlled places, a CLI as the reference client, and a deterministic way to project scoped direction into the work that needs it.

Candidate and durable records live in a tracked ledger. Live records can coordinate active work through shared Git runtime state. Generated Markdown exists for humans, but it is not the source of truth.

The protocol is the important part. The CLI is only the first client.

I wanted Codex, Claude, IDE agents, CI tools, review systems, and future agent platforms to be able to point at the same repository decision graph without one provider owning the memory layer.

Provider objects stay opaque. A message is still owned by the provider. A PR is still owned by GitHub. A commit is still Git. Fabric keeps references and relations, not private transcripts.

## What The First Version Does

The first version can initialize a repository, start scoped threads, record notes, run preflight projections, sync pending direction, acknowledge provider exposure, add causal relations, explain a node, challenge older direction, and curate records after work completes.

A simple flow looks like this:

```bash
fabric thread start --issue VS-123 --area virtual-store/listing \
  --path 'internal/listing/**'

fabric preflight "add listing filters" --issue VS-123 \
  --area virtual-store/listing --path 'internal/listing/**'

fabric note --candidate --reason "The shared endpoint owns pagination semantics" \
  "Extend the existing listing endpoint instead of creating another one."

fabric sync
```

The command syntax is not the interesting part.

The interesting part is the loop.

A correction happens once. It becomes a compact scoped record. A related thread receives it before acting. Later, if an action or commit implements it, that relationship can be recorded explicitly.

Now the repository can answer a better question than Git alone usually can:

Not just “what changed?”

But:

What direction was available before this change?

What evidence supported it?

Was it merely exposed to the model, or did the action explicitly claim it was causal?

Which rejected path was this supposed to avoid?

## Why Immutable Events Matter

Parallel work also changed how I thought about updates.

If multiple worktrees can write direction, the system cannot silently resolve conflicts by picking the newest file. That would hide the exact disagreement it should surface.

So every mutation is an immutable event.

A record is created once. Lifecycle changes append child events. If two lifecycle changes extend the same parent, that is a conflict. A reader reports the competing children instead of picking a winner by timestamp, file order, or ID order.

This matters because disagreement is not a storage bug.

If one thread says old direction is expired and another says it is still active, that is repository state. A human may need to decide. The system should not quietly paper over it.

## What I Do Not Want This To Become

I do not want this to become a transcript archive.

That would recreate the same mess in a new format.

I do not want it to become an AI summarizer that extracts “decisions” from every conversation and fills the repository with junk.

That would make the project worse.

I do not want it to become a provider-specific memory plugin.

The whole point is that repository direction should outlive whichever agent tool produced or consumed it.

And I do not want it to become an orchestration platform too early.

There is a world where this direction layer feeds orchestration, review, merge planning, or agent-worker systems. But those are higher layers. Worktrees, cloud sandboxes, remote agents, and CI jobs are execution details.

The core should stay smaller:

Repository direction should be scoped, lifecycle-aware, portable, and causally traceable.

## The Hard Part Is Capture

The hardest part is not writing event files.

The hard part is deciding what deserves to become repository direction.

Most comments should not. Most agent observations should not. Some review feedback is too specific. Some findings are wrong. Some constraints expire. Some rules are really preferences.

So capture has to be sparse and reviewable.

Agents can propose candidate records. Tools can mine PRs. Review workflows can suggest reusable direction. But durable promotion should stay human- or reviewer-confirmed.

The system should make it cheap to preserve useful reasoning.

It should not turn every sentence into law.

## What I Learned

The biggest thing I learned is that “agent memory” is too broad for the problem I care about.

Remembering more is not automatically useful.

In software engineering, the useful memory is often small, scoped, and normative. It is not only something that happened. It is something that should constrain, guide, explain, or challenge future work.

The second thing I learned is that a good system should be comfortable saying “unknown.”

If something was delivered, say delivered.

If something was exposed, say exposed.

If something explicitly informed an action, say informed.

If all you have is a guess, do not launder it into provenance.

The third thing I learned is that local-first is not just a distribution choice. It is a trust boundary.

If repository direction matters, the repository should be able to own it. Teams should be able to inspect it, commit it, review it, branch it, merge it, and keep using it even if agent providers change.

That is the kind of boring infrastructure I trust.

## Where This Goes

The next step is not to make the system remember everything.

It is to make the capture loop feel natural at the moment useful direction appears.

When a reviewer corrects an agent, preserving the reusable part should be almost as easy as leaving the comment itself. When a sibling thread starts related work, receiving that correction should be automatic. When someone asks why a commit happened, the answer should distinguish what was available from what was causal.

The thing I want to prove is simple:

Can a thread avoid a bad path because another thread already paid to discover it?

If yes, this is worth continuing.

Because the goal is not to make agents sound like they remember the repository.

The goal is to stop treating every new thread like the repository’s first day.
