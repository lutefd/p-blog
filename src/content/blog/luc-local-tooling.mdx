Luc is a terminal AI agent built around a streaming TUI, local tool execution, and runtime extension points.

The interesting part is not just that it can chat with a model. It is the amount of surface area that stays local and configurable: provider config, keychain credentials, workspace overrides, built-in tools, runtime providers, hooks, prompts, skills, themes, packages, and UI extensions.

## A Terminal UI With Real Controls

The README describes a practical TUI loop: start a session with `luc`, resume one with `luc open <id>`, inspect provider/workspace health with `luc doctor`, reload extensions without restarting, or use `luc rpc` for machine-readable JSON mode.

The keybindings are also product decisions. There are shortcuts for the inspector pane, command palette, model switching, session list, extension reload, and stopping a turn. That says Luc is trying to be an everyday tool, not just a demo wrapper around an API.

## Provider Configuration Should Be Boring

Luc reads user-global config from `~/.config/luc/config.yaml` and `~/.luc/config.yaml`, then merges workspace-local `.luc/config.yaml` on top. The minimal provider shape is OpenAI-compatible: base URL, API key environment variable, and model.

Credentials can come from the environment or the OS keychain. That detail matters because agent tools become annoying fast when every workspace needs a custom secret ritual.

## Runtime Extensions

The extension model is the real spine of the project. Luc supports tools, long-lived extension hosts, async hooks, custom providers, UI commands and views, prompt extensions, skills, and themes under `~/.luc` or workspace-local `.luc`.

Custom providers can be OpenAI-compatible endpoints or exec adapters that stream JSONL provider events. Shared packages can be installed from GitHub or local paths.

The shape is clear: Luc wants the agent runtime to be programmable without recompiling the binary.

## The Product Bet

The bet is that a terminal agent should be close to the files and commands it changes, but extensible enough to adapt to a team's workflow. That is a good place to build from.
