Fabric starts from an annoying failure mode: the repository already learned something, but the next thread does not inherit the learning.

A reviewer explains why a behavior belongs at a shared boundary. A PR changes direction. The code merges. Weeks later, another agent working in a different package proposes the same locally reasonable shortcut, because the old review never became reusable project direction.

That is not only a prompting problem. It is a memory-shape problem.

## The Patch Survives, The Reason Becomes Folklore

Software teams already spend the expensive human time. They debate boundaries, reject shortcuts, explain incidents, and write review comments. The final diff remains in Git, but the reason often gets trapped in a PR, a transcript, or a maintainer's memory.

Fabric treats the reusable unit as a compact record: a decision, finding, requirement, challenge, or review direction with scope, rationale, trust, lifecycle, and evidence references. The record is small on purpose. It is not trying to archive the whole conversation.

## Scope Is The Point

Large projects do not have one global truth. Routing, caching, build tooling, data boundaries, and product workflows each have local constraints. A decision that matters deeply in one area may be noise everywhere else.

Fabric records can be scoped by repository, issue, pull request, configured area, or path. A thread declares what it is working on, and deterministic projection selects the relevant active records within a budget. That keeps memory from becoming another giant instruction file.

## Availability Is Not Causality

One of the sharper ideas in Fabric is the split between availability and causal influence.

A projection can prove that a record was selected and delivered. A provider adapter can confirm that it entered model context. That still does not prove the agent used it.

For "why did this happen?" views to be honest, actions and commits need explicit causal relations such as `informed_by` or `implements`. Fabric refuses to infer motivation from text similarity or from context delivery alone.

## Local First, Provider Neutral

The Local V1 design is intentionally modest: immutable event files, Git-backed ledger state, generated human views, and shared Git-common runtime state for sibling worktrees. It works without an account, daemon, network service, database, or model call.

Provider objects stay opaque. Codex, IDE agents, CI systems, and future tools can point at messages, actions, commits, issues, or PRs without copying private transcripts into repository memory.

## What I Like About The Shape

Fabric is not trying to make agents remember everything. It is trying to make repositories remember the few things that should change future work.

That distinction matters. A transcript is what happened. A Fabric record is what the next person or agent should not have to rediscover.
