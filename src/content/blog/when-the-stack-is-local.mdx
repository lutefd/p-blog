The problem with stacked branches is not the stack.

The problem is everything around it.

In a small repository, you can usually keep the shape in your head. This branch depends on that branch. This PR should land first. That one is just a cleanup. The integration branch is temporary. The test environment is running the right combination of changes. Everyone involved knows the order.

Then the repository gets large.

The branches multiply. The PRs stop being a neat staircase. Some work depends on another branch, but not in the exact way a stacked PR tool expects. Some branches exist only to get a test environment into the right state. Some PRs are reviewable on their own, but only meaningful when composed with three others. Some have already been rebased twice. Some are waiting on a backend change. Some are waiting on a frontend change. Some are waiting on the one PR everyone forgot was still in the middle.

At that point, the stack is no longer a clean diagram.

It is a local truth that keeps leaking out of your head.

## The Stack That Was Not Quite A Stack

The awkward part was that the work really was stacked.

Not philosophically. Practically.

One branch needed another branch's API shape. Another branch needed the shared types. Another needed the feature flag. Another needed the database migration. The end-to-end test environment needed all of them together, even though review still needed them split apart.

But the team did not follow stacked PRs bar for bar.

There was no single hosted workflow where each PR cleanly pointed to the previous one and the whole thing could be understood from remote metadata. The real dependencies were sometimes local, sometimes temporary, sometimes shaped by test environments, and sometimes different from the review graph.

So the dependency graph lived in human memory.

That is already bad.

Then the graph started changing.

## Integration Branches As Survival Tools

The review branches were not enough because validation needed a composed state.

If you want to run end-to-end tests against a test environment, you often need the whole feature path assembled: backend contracts, UI changes, config, flags, fixes, and whatever small compatibility branch makes the rest of it work.

So integration branches became a survival tool.

Take these branches.

Put them on top of this base.

Create a branch that represents the thing we need to deploy to the test environment.

Run the tests there.

Then do it again after someone rebases, after another PR changes, after the base branch moves, after a conflict gets resolved, after a small fix lands in the middle.

This is exactly the sort of workflow Git can technically support and humans can technically execute.

Technically.

The problem is that "technically" does a lot of damage in a large repo.

## The Failure Modes Were Boring And Expensive

Nothing about this failed in a cinematic way.

It failed in the boring ways that waste afternoons.

A PR was forgotten in the middle of the chain. The integration branch looked fine until someone realized the test environment was missing a small behavior that only existed in that forgotten branch.

A conflict was resolved quickly because everyone had context fatigue. The code compiled. The diff looked acceptable. Later, an end-to-end test failed because a small but critical line had been deleted during conflict resolution.

A branch was rebased onto the wrong parent. Not obviously wrong. Just wrong enough that another branch now carried duplicated changes, or missed a dependency, or made review harder than it needed to be.

Someone recreated an integration branch manually but skipped one of the pieces because the dependency lived in a Slack message, a PR comment, or someone's memory.

Someone else fixed the same conflict in a slightly different way.

Then the team had to answer the worst possible Git question:

What exactly is in this branch?

Not "what commits are in it?" Git can answer that.

What is this branch supposed to represent?

Which source branches were composed into it?

Which dependencies were intentional?

Which branch is missing?

Which conflict resolution changed behavior?

Which state was deployed to the test environment when the E2E run passed?

That is the chaos. Not Git itself. The missing orchestration state around Git.

## Remote Metadata Was The Wrong Source Of Truth

It is tempting to solve this by making the remote platform smarter.

Stacked PR metadata. Labels. Naming conventions. PR descriptions. Checklists. Comments. Branch protection. Automation.

All of that can help.

But it still felt like the wrong source of truth for this particular problem.

The branches are local before they are PRs. The dependency relationship matters before review is opened. The integration branch may exist only to deploy a test environment. The compose recipe may need to change faster than the PR metadata. And sometimes the review graph and the integration graph are simply not the same graph.

I wanted the local repository to know the shape.

Not GitHub.

Not a convention hidden in PR titles.

Not a checklist someone had to remember to update.

The local repo.

## The Missing Primitive Was A Declared Dependency

Once I stopped trying to make the whole workflow look like a hosted stacked PR system, the abstraction got smaller.

I needed a way to say:

This branch depends on that branch.

This branch should be synced after its parent.

This group of branches should be composed onto this base.

This integration recipe should be saved because I will need it again.

This generated integration branch came from that recipe and should be easy to list or delete later.

That is much less ambitious than "manage all review."

It is local orchestration state.

The important part is that the dependency graph becomes explicit. Not inferred from branch names. Not reconstructed from PRs. Not held in someone's head.

Declared.

## Then It Became Weaver

That is when the tool became Weaver.

Weaver is a local-first CLI for managing stacked Git branches without requiring GitHub metadata, PR conventions, or external services.

It stores branch relationships in `.git/weaver/`, keeps them out of the committed tree, and resolves the stack as a DAG. `weaver stack` declares dependencies. `weaver deps`, `weaver status`, and `weaver doctor` make the shape inspectable. `weaver update` refreshes branches from upstream. `weaver sync` brings a stack back into dependency order. `weaver compose` creates an ephemeral or materialized integration state from a set of branches.

The command list is not the interesting part.

The loop is.

Declare the dependencies.

Inspect the graph.

Sync in dependency order.

Compose the branches needed for validation.

Save integration recipes when they are reused.

Create or update an integration branch when a test environment needs something deployable.

That is the workflow I kept doing manually.

The tool just makes the shape explicit enough that it can be repeated.

## Compose Is Not Sync

One of the design details I care about is that the commands do not pretend every operation is the same.

Refreshing a branch from its upstream is not the same as syncing it with its parent.

Syncing a stack is not the same as composing a deployable integration branch.

Reviewing a PR is not the same as validating the combined feature in a test environment.

So the tool separates those jobs.

`weaver update` fetches and fast-forwards local branches from their configured upstream refs. It does not decide how stack parents should flow into children.

`weaver sync` applies dependency order. By default it can keep the clean stacked-diff workflow with rebase, but merge-based sync exists for branches that already have open PRs, review comments, or other consumers where rewriting history would be painful.

`weaver compose` builds the combined state. By default it is ephemeral. When needed, it can create or update an integration branch. Saved integration strategies live separately, because "these branches together on this base" is its own piece of workflow state.

That separation came directly from the chaos.

When those actions blur together, people make dangerous assumptions.

## Safety Is Mostly Remembering What Happened

The scary part of this workflow is not that Git can merge badly.

The scary part is not knowing which merge happened.

A conflict resolution can delete one small critical piece of code. A compose attempt can skip a branch. A sync can leave the repository on a different branch than the one you thought you were working from. A broken integration branch can stay around and look more official than it is.

So the safety model had to be boring and explicit.

Mutating Git commands are printed before execution.

Sync state is persisted before each step.

`weaver abort` restores the original branch.

Compose is ephemeral by default.

Compose failures report the branch that failed and the conflicting files.

Integration branches created by compose can be listed and deleted later.

Saved integrations can be inspected and diagnosed.

None of this makes Git conflict-free. That would be fantasy.

It makes the workflow less dependent on memory when conflicts happen.

## What I Learned

The first thing I learned is that stacked work is not always a PR feature.

Sometimes it is a local coordination problem that happens to produce PRs.

That distinction matters. If the stack only exists in the review platform, you are forced to make every local dependency fit the hosted model. But large-repo feature work often has a messier shape: review branches, test-environment branches, temporary compatibility branches, and integration recipes that exist for validation more than review.

The second thing I learned is that integration branches are real artifacts even when nobody wants to admit it.

Teams create them because they need deployable states for testing. Pretending they are ad hoc does not make them safer. It just means the recipe is harder to repeat and easier to forget.

The third thing I learned is that the dangerous failures are usually small.

The forgotten PR.

The skipped branch.

The conflict resolution that removed one line.

The integration branch that was almost right.

Those are the failures a local tool can actually help with, not by being clever, but by keeping the graph and recipe visible.

## Where This Goes

I do not want this kind of tool to hide Git.

That would be the wrong lesson.

The point is to make the work around Git inspectable: the declared dependencies, the compose order, the integration recipes, the temporary branches, the recovery state.

The ideal version does not make branching feel magical. It makes the manual workflow less lossy.

Can a large-repo feature stack be split for review while still being composed reliably for validation?

Can a test environment branch be recreated without trusting someone's memory?

Can a forgotten PR become visible before it breaks the deployable state?

Can conflict recovery preserve enough context that one small critical deletion does not slip through silently?

That is the problem I wanted to solve.

Not stacked PRs as a perfect ideology.

Stacked work as it actually happens when the repository is large, the branches are many, and the test environment needs the whole thing today.
